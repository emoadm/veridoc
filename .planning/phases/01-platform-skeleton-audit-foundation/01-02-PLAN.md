<objective>
Build veridoc-audit, the shared tamper-evident audit SDK (PLAT-02, D-04/D-05/D-06)
that every later service writes through. It computes a per-record hash chain
`SHA-256(prev_hash || JCS(payload))` over RFC-8785-canonicalized payloads, inserts
append-only rows inside the CALLER's open transaction (so business + audit commit
atomically), serializes chain-head access with a Postgres advisory lock to prevent
forks, and ships an append-only DB schema (UPDATE/DELETE trigger) carrying agent
decision/confidence columns now to avoid migrating an immutable table later.

Purpose: This is the single most correctness-critical component of the phase. A
subtle bug here (non-deterministic serialization → false tamper positives; separate
transaction → unrecorded actions; forked chain) invalidates the entire 21 CFR Part 11
audit posture. Built once, cloned by every later phase.
Output: A tested veridoc-audit lib with JCS canonicalization, chain compute/verify,
same-transaction writer, and the four Wave 0 audit tests green.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| business handler → audit_log (DB) | Every state change must produce an immutable record in the same transaction |
| DBA/privileged SQL → audit_log | A privileged actor attempting to rewrite history is the core threat |
| concurrent writers → chain head | Racing writers could fork the chain |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering / Repudiation | audit_log row rewritten or deleted | mitigate | Hash chain (compute_record_hash) + BEFORE UPDATE OR DELETE trigger + INSERT/SELECT-only grant; test_tamper_detection re-walk proves detection |
| T-02-02 | Tampering | non-deterministic canonicalization → false tamper positives masking real tampering | mitigate | RFC 8785 JCS + committed golden-vector test (Pitfall 2) |
| T-02-03 | Tampering | forked chain under concurrency (two rows, same prev_hash) | mitigate | pg_advisory_xact_lock serializes chain-head read-modify-write (Pitfall 1) |
| T-02-04 | Repudiation | business action commits without its audit row (separate txn) | mitigate | append_audit joins caller's transaction, no internal commit; test_same_txn forces audit failure → business rollback (Pitfall 5) |
| T-02-05 | Tampering | SQL injection into audit payload | mitigate | SQLAlchemy parameterized inserts only; payload stored as jsonb, never string-formatted SQL |
| T-02-06 | Information Disclosure | PII written in plaintext into audit before/after jsonb | accept | Audit before/after carry already-pseudonymized/encrypted values from callers (plan 03 helpers); audit SDK is value-agnostic — documented contract for plan 05 |
</threat_model>
