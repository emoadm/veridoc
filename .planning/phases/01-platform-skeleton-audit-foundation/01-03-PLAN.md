<objective>
Build veridoc-crypto + veridoc-pseudonym TOGETHER under one per-patient key
hierarchy (PLAT-03 Success Criterion #4, D-11 + D-12). veridoc-crypto provides
app-level AES-256-GCM envelope encryption behind a provider-portable KMS abstraction
(AWS KMS / Azure Key Vault; local keyring for tests — NOT pgcrypto). veridoc-pseudonym
provides deterministic patient tokens. Both derive from a master key + HKDF-derived
PER-PATIENT key so that GDPR right-to-erasure = deleting one patient's derivation
material, which simultaneously makes that patient's pseudonym irrecomputable and
ciphertext undecryptable WITHOUT touching other patients or the append-only audit trail.

Purpose: D-11 (encryption) and D-12 (deterministic pseudonym + erasure-by-key-deletion)
are mutually exclusive if designed separately with a global key (RESEARCH Pitfall 3,
assumption A7 HIGH RISK). They must be designed as one key hierarchy. This is the second
most correctness-critical design in the phase.
Output: Tested veridoc-crypto + veridoc-pseudonym libs with field-encryption,
determinism, and crypto-shred tests green, plus a tracked KEY-HIERARCHY.md.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| application → KMS/HSM | DEKs are wrapped/unwrapped here; plaintext keys never persist in the DB engine |
| DB at rest → attacker | Stored PII columns must be ciphertext, never plaintext |
| data subject erasure request → key store | Deleting one patient's key must not affect others |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Information Disclosure | PII readable at rest | mitigate | AES-256-GCM envelope encryption (D-11); test_field_encryption asserts ciphertext ≠ plaintext |
| T-03-02 | Information Disclosure / Compliance | non-erasable PII (global key) breaks GDPR Art. 17 | mitigate | Per-patient HKDF key hierarchy; erase_patient isolates erasure; test_crypto_shred proves A erased, B intact (Pitfall 3, A7) |
| T-03-03 | Tampering / Information Disclosure | hand-rolled AES (nonce reuse, missing AAD) | mitigate | Use vetted AEAD library (Tink / AWS Enc SDK) per Task 1; never hand-roll crypto (Don't Hand-Roll) |
| T-03-04 | Information Disclosure | plaintext DEK or key in the DB | mitigate | DEK wrapped by KMS abstraction; only wrapped DEK + ciphertext stored; keys out of DB engine (no pgcrypto) |
| T-03-05 | Information Disclosure | deterministic pseudonym enables re-identification by linkage | accept | Determinism is required for cross-source SDV matching (D-12); token derived from secret per-patient key (HMAC) — not reversible without the key; erasure via key deletion |
| T-03-06 | Spoofing | cloud-provider lock-in via non-portable KMS | mitigate | Provider-portable KMS interface (AWS KMS / Azure KV); LocalKeyring for tests; DEC-cloud-provider stays open |
</threat_model>
