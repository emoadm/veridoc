<objective>
Wire the ONE reference service end-to-end (D-07 walking skeleton, PLAT-01/02/03):
HTTP → Keycloak authn/authz (veridoc-auth) → tenancy context (veridoc-tenancy) →
a thin business write (register/update a tenancy-scoped synthetic "Subject") that
envelope-encrypts the PII field (veridoc-crypto), derives the deterministic pseudonym
token (veridoc-pseudonym), and writes a same-transaction hash-chained audit row
(veridoc-audit) → encrypted Postgres persistence. Every cross-cutting concern is
proven exactly once against a live Keycloak + Postgres so Phases 2–7 clone this exact
path. Also produce the Dockerfile that plan 06 packages.

Purpose: This is the integration point where all five shared libs meet. The reference
business action (RESEARCH Open Question #3, left to discretion) is deliberately thin —
a tenancy-scoped Subject create+update — chosen because it exercises authn → authz →
tenancy → audited write → encrypted persistence in one path without pulling in any
later-phase domain logic.
Output: A running FastAPI reference service + Dockerfile + integration tests (RBAC,
login-attempt audit, field-encryption-at-rest) green against live Keycloak + Postgres.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → reference service (HTTPS) | All cross-cutting controls converge at this entry |
| service → Postgres | Business + audit + encrypted PII persist here in one transaction |
| service → Keycloak (JWKS) | Token trust anchor |
| service → KMS | DEK wrap/unwrap for PII |
| erasure request → key store vs audit_log | Crypto-shredding one patient's key must NOT mutate/delete append-only audit rows (Art.17 erasure must not break Part 11 immutability) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Repudiation / Tampering | business write without atomic audit | mitigate | append_audit in the same session/transaction; verify_chain asserted in tests (D-05) |
| T-05-02 | Information Disclosure | secrets baked into image / config | mitigate | No secrets in Dockerfile; config via env/K8s Secrets (plan 06); grep gate in Task 3 verify |
| T-05-03 | Elevation / Information Disclosure | cross-role or cross-tenant access | mitigate | require_role deny-by-default + fail-closed current_tenant; test_rbac asserts 403 + cross-tenant denial |
| T-05-04 | Information Disclosure | PII persisted in plaintext | mitigate | encrypt_field before persist; test_field_encryption asserts ciphertext ≠ plaintext at rest |
| T-05-05 | Spoofing | MFA bypass at the service edge | mitigate | authn_dependency enforces acr/amr (plan 04); reference service mounts it on protected routes |
| T-05-06 | Tampering | SQL injection / unvalidated input | mitigate | Pydantic v2 input models + SQLAlchemy parameterized queries |
| T-05-07 | Denial of Service | unauthenticated flood on /subjects | accept | Rate-limit hook deferred; single-cluster milestone; /healthz only unauthenticated route |
| T-05-08 | Tampering / Compliance | GDPR Art.17 erasure mutating or deleting append-only audit rows (breaking 21 CFR Part 11 immutability), OR erasure failing to render the audited PII undecryptable | mitigate | Crypto-shred deletes only the per-patient key, never audit rows; test_erasure_audit_immutability asserts verify_chain still True (rows untouched) AND A's audited ciphertext undecryptable AND B intact |
